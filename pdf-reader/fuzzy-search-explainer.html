How fuzzy search works in this PDF reader

This app uses Levenshtein distance over token windows. The idea: split text into tokens (words), slide a window over them, compare each window to the query, and keep windows that are close enough.

1) Normalize and tokenize

Both the query and PDF text are normalized: lowercased and stripped of punctuation noise. Then text is tokenized into words. 

"The Encyclopædia Britannica, 1911" -> ["the", "encyclopædia", "britannica", "1911"]

2) Sliding token windows

For a query with k words, we slide a window of length k across page tokens. Each window is a candidate match.

QuerykWindow examples
neural network2["neural","netwrok"], ["deep","network"], ...

3) Score each window with Levenshtein distance

Join query tokens and window tokens with spaces, then compute edit distance. Lower distance means more similar. 

distance("neural network", "neural netwrok") = 2

If the distance is <= the user-selected threshold ("Max distance"), the window is a match.

4) Rank and navigate

Matches are sorted by:

  1. smaller distance first,
  2. then earlier page number,
  3. then earlier position on page.

The app lets users jump through these ranked matches and highlights them in the PDF text layer.

How partial matching works (single-word query)

For one-word queries, the app also compares the query against substrings of each token, so enc can match encyclopedia even without a space boundary.

How to extend this for multiple-word fuzzy search

Current token windows already support multiple words. To make it stronger, you can add:

  1. Variable window sizes (e.g. query length ± 1) to tolerate missing/extra words.
  2. Per-token distance + alignment (dynamic programming over token sequences), instead of only string-level distance on joined text.
  3. N-gram indexing to prune candidates before expensive Levenshtein checks.
  4. Weighted costs (smaller penalties for keyboard-neighbor typos/transpositions).
  5. Semantic expansion (stemming/synonyms) before fuzzy scoring.
Practical upgrade path: keep the existing window approach, but score each candidate as a weighted combination of (a) token-level edit distance, (b) word-order penalty, and (c) coverage of query terms. This works well for multi-word queries with insertions/deletions.

Pseudocode

tokens = normalize(pdfPageText).splitWords()
q = normalize(query).splitWords()

for each page:
  for each window in slidingWindows(tokens, length = len(q)):
    score = levenshtein(join(q, " "), join(window, " "))
    if score <= threshold:
      collect(page, windowStart, score)

sort(matches, by score, then page, then position)
